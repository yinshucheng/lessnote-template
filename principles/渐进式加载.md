# 渐进式加载（Progressive Loading）

> 上下文工程的核心实践：不要一次性把所有信息塞进上下文窗口，而是分层组织、按需加载。

## 核心洞察

渐进式加载**不是某个框架的专属特性**，而是一种通用的上下文工程模式。它的本质是：

```
信息按「访问频率 × 重要性」分层 → 轻量索引常驻 → 重内容按需加载
```

适用于：代码库、知识库、个人笔记、文档系统——任何需要 AI 在有限上下文中工作的场景。

## 各系统实现对比

### 1. Claude Code / OpenCode — 多层级文件体系

| 层级 | 内容 | 加载时机 | Token 开销 |
|------|------|----------|-----------|
| L0 系统提示 | 基础指令 + 环境信息 | 每轮对话 | 固定 |
| L1 CLAUDE.md | 项目规则、约定、路由表 | 每轮对话（常驻） | 中等，建议精简 |
| L2 Skill 索引 | 技能名 + 一句话描述 | 每轮对话（嵌入 tool 定义） | 小 |
| L3 Skill 内容 | 完整 SKILL.md 工作流 | 用户 `/command` 或 AI 判断触发 | 按需，50-350 行 |
| L4 目录级 CLAUDE.md | 子目录特定规则 | 读取该目录文件时自动注入 | 自动按需 |
| L5 文件内容 | 源代码/文档全文 | AI 主动调用 read 工具 | 按需 |

**关键设计**：
- L1 是路由层——告诉 AI「遇到 X 场景去读 Y」，而不是把 Y 的全部内容放进来
- L4 是位置感知的——进入某个子目录，相关规则自动生效
- `<system-reminder>` 标签作为运行时注入机制

### 2. OpenClaw — Bootstrap + Skills + 上下文压缩

| 层级 | 内容 | 加载时机 | Token 管理 |
|------|------|----------|-----------|
| Bootstrap 文件 | AGENTS.md, SOUL.md, TOOLS.md 等 | 每轮，截断（头 70% + 尾 20%） | 总上限 150K 字符 |
| Skills 列表 | 名称 + 描述 + 路径 | 每轮（紧凑格式） | 上限 30K 字符 |
| Skill 指令 | 完整 SKILL.md | AI 判断后用 `read` 工具加载 | 按需 |
| 记忆文件 | memory/*.md | AI 用 memory_search 检索 | 按需 |

**独特机制**：
- **上下文压缩**（Context Pruning）：软裁剪（保留头尾各 1500 字符）→ 硬清除（替换为占位符）
- **运行模式降级**：子 Agent 用 `minimal` 模式，省略 Skills/Memory 等非必需部分
- **eligibility 过滤**：根据运行环境（OS、已安装工具）在列表阶段就排除不相关 Skill

### 3. NanoClaw — 双系统 Skill 架构

| 层级 | 内容 | 加载时机 |
|------|------|----------|
| CLAUDE.md 路由表 | Skill 名 + 一句话「何时用」 | 常驻 |
| SKILL.md 指令 | 完整工作流（50-350 行） | `/command` 触发 |
| Skill 内资源 | references/*.md, resources/*.md | SKILL.md 引用时再读取 |
| Skills Engine | manifest.yaml + 三路合并 | SKILL.md 指示执行时 |

**特色**：Skill 内还有二级延迟加载——SKILL.md 引用的参考文档只在执行过程中按需读取。

### 4. Aider — Repo Map（图排序索引）

| 层级 | 内容 | 加载时机 |
|------|------|----------|
| Repo Map | 全库符号的结构化摘要（PageRank 排序） | 每轮，Token 预算内 |
| Read-Only 文件 | 参考文件全文 | 用户指定 |
| In-Chat 文件 | 编辑目标文件全文 | 用户指定 / AI 选择 |

**独特机制**：
- Tree-sitter 解析全库符号 → 构建文件间引用图 → PageRank 排名
- **动态预算**：无文件在 Chat 时，Repo Map 预算扩大 8 倍；添加文件后自动缩小
- **ContextCoder**：用 LLM 自身来决定哪些文件需要加载到上下文

### 5. Serena — 符号级渐进钻取

| 层级 | 工具 | 返回内容 |
|------|------|----------|
| L0 文件列表 | list_dir / find_file | 文件名 |
| L1 符号概览 | get_symbols_overview(depth=0) | 符号名 + 类型 |
| L2 成员结构 | get_symbols_overview(depth=1) | 类的方法列表 |
| L3 符号定位 | find_symbol(include_body=False) | 位置 + 签名 |
| L4 文档信息 | find_symbol(include_info=True) | docstring + 类型 |
| L5 完整源码 | find_symbol(include_body=True) | 全部代码 |
| L6 引用关系 | find_referencing_symbols | 调用方 + 代码片段 |

**核心原则**：名称优先 → 结构其次 → 内容最后。每一层都让 AI 决定是否需要更深入。

## 通用设计模式提取

### 模式 1：索引-内容分离（Index-Content Separation）

```
常驻层：轻量索引（名称 + 一句话描述 + 路径/标识符）
按需层：完整内容（只在索引命中时加载）
```

所有系统都遵循这个模式：
- Claude Code：Skill 名 → SKILL.md
- OpenClaw：Skills 列表 → `read` Skill 文件
- Aider：Repo Map 符号名 → 文件全文
- Serena：符号概览 → 符号体

### 模式 2：路由表驱动（Router Table）

在常驻上下文中放一张路由表，格式：

```
| 场景/触发条件 | 去哪读 | 为什么 |
```

AI 根据当前任务匹配路由表，决定加载什么。不需要把所有内容放进上下文，只需要让 AI 知道「什么东西存在」和「在哪找」。

### 模式 3：位置感知注入（Location-Aware Injection）

进入某个「区域」时，自动加载该区域的上下文：
- Claude Code：读取子目录文件时自动注入目录级 CLAUDE.md
- 知识库场景：打开某个主题时自动加载相关背景

### 模式 4：多级粒度钻取（Multi-Level Drill-Down）

```
概览 → 结构 → 摘要 → 全文
```

不是二元的「加载/不加载」，而是提供多个粒度级别，让 AI 逐步深入。

### 模式 5：预算感知裁剪（Budget-Aware Trimming）

- Aider：Token 预算内的 binary search 找最大可展示内容
- OpenClaw：头尾保留策略的软/硬裁剪
- Serena：max_answer_chars 参数限制每次返回量

## LessNote 知识库实践方案

> 以下是将渐进式加载应用于个人知识库（LessNote）的具体方案。

### 当前问题

LessNote 的 CLAUDE.md 已经 230 行，包含：身份定义、认知偏见、系统集成、工作哲学、项目原则……所有内容每轮对话都全量加载，不管当前任务是「写日记」还是「做技术决策」。

### 方案：三层渐进架构

#### 第一层：常驻路由表（CLAUDE.md 精简版）

CLAUDE.md 只保留：
1. **核心身份**（5 行）——不可省略的人格定义
2. **路由表**——场景 → 文档路径的映射
3. **绝对规则**（3-5 条）——无论什么场景都适用的硬约束

```markdown
# LessNote

## 核心身份
说到做到，hardcore，self-discipline。行动产生信息。

## 上下文路由表

| 场景 | 读取 |
|------|------|
| 每日启动/规划 | goals/人生目标与原则参考.md |
| 认知偏见检测 | goals/认知误区与陷阱.md |
| 行为问题诊断 | principles/福格行为模型.md |
| 结构化决策 | principles/结构性思考框架.md |
| 情绪低落/反思 | journal/ 近期日记 |
| 涉及他人 | life/社会关系维护.md |
| 工作决策 | life/美团业务背景.md |
| 技术方案选型 | ai-prompts/圆桌讨论-李继刚.md |
| 概念理解 | ai-prompts/撕考者-李继刚.md |
| 深度思考 | ai-prompts/沉思者-李继刚.md |
| 周/月回顾 | goals/人生目标与原则参考.md + goals/2026年目标.md |

## 硬约束
- 小红书本周发了没？（每次 /morning 必问）
- 22:00 后禁止新番茄
- 文档修改后立即 git commit
```

从 230 行 → ~30 行。省下的 Token 让 AI 有更多空间处理实际任务。

#### 第二层：场景文档（按需加载）

现有的 `goals/`, `principles/`, `life/`, `ai-prompts/` 目录已经天然形成了按场景组织的文档。不需要改结构，只需要让路由表指向它们。

**增强**：给每个目录加一个 `CLAUDE.md`（目录级指令），定义该区域的交互规则。例如：

```
# journal/CLAUDE.md
读取日记时：
- 关注情绪模式，不要评判
- 如果发现连续低落，主动关联 goals/认知误区与陷阱.md
```

```
# principles/CLAUDE.md
读取思考框架时：
- 必须要求用户给出具体案例来应用
- 不要停在「理解框架」层面，必须产出行动
```

#### 第三层：Skills（工作流封装）

现有的 `/morning`, `/evening`, `/focus`, `/checkpoint` 已经是 Skill 了。进一步的做法：

1. **Skill 内引用路由表**——Skill 不内联所有内容，而是根据当前情况动态读取

```markdown
# /morning SKILL.md
...
步骤 3：检查今日 TOP 3 是否清晰
- 如果某任务方向不清 → 读取 ai-prompts/撕考者-李继刚.md，用撕考框架拆解
- 如果是技术选型 → 读取 ai-prompts/圆桌讨论-李继刚.md
...
```

2. **新增 Skill 候选**：

| Skill | 场景 | 按需加载 |
|-------|------|----------|
| `/decide` | 面临选择时 | principles/结构性思考框架.md |
| `/diagnose` | 拖延/做不到时 | principles/福格行为模型.md |
| `/reflect` | 情绪处理/反思 | journal/ 近期 + goals/认知误区与陷阱.md |
| `/people` | 涉及他人沟通/决策 | life/社会关系维护.md |

### 实施路径

```
Phase 1（30 分钟）：精简 CLAUDE.md → 路由表模式
Phase 2（可选）：给关键目录加 CLAUDE.md
Phase 3（可选）：新 Skills 封装常用工作流
```

Phase 1 是最高 ROI 的改动——只改一个文件，立即减少每轮对话的 Token 浪费，同时不丢失任何能力（AI 需要时仍然可以通过路由表找到所有信息）。
